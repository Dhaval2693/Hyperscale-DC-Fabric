# BGP Path Selection: How BGP Chooses the Best Path

BGP is not just a reachability protocol. It is a policy engine. And the mechanism through which policy is enforced is the **best path selection algorithm** — the ordered set of rules BGP applies every time it has more than one path to the same destination.

Unlike OSPF or IS-IS, which use a single metric to find the shortest path, BGP receives route information with a rich set of **path attributes** attached. When BGP learns the same prefix from multiple neighbors, it compares those attributes in a fixed priority order. The first attribute where the competing paths differ determines the winner. If two paths are equal on every attribute all the way down, BGP falls back to final tiebreakers. Understanding this order is not just academic — it is the foundation of every BGP traffic engineering decision.

There are eleven steps in the algorithm. The first six are the ones engineers actively control. The rest are tiebreakers the protocol uses when no policy has been expressed.

**Weight** is evaluated first. It is a Cisco-proprietary attribute, local to the router — it is never advertised to any neighbor. Weight has a default of 0, and the **highest** value wins. Because it is not shared, it is the sharpest tool available when you need to influence path selection on a single router without affecting anything else in the network. If two paths arrive on the same router and you want one preferred without telling the rest of the AS, you set Weight.

**Local Preference** comes next. Unlike Weight, Local Preference is shared among all iBGP routers in the AS — configure it on one router, and every router in the AS sees it. The default is 100, and the **highest** value wins. Local Preference is the standard tool for controlling **outbound** traffic: which exit point the AS uses to reach a destination. If you want all traffic to a particular prefix to leave through ISP A rather than ISP B, you set a higher Local Preference on the routes learned from ISP A. Every router in the AS will agree on that preference without needing individual configuration.

Next, BGP prefers routes that the local router itself originated. Before a router can advertise any prefix, it must first inject it into BGP — and there are three ways to do this. The **network command** matches an exact prefix already present in the routing table and injects it into BGP; the prefix must exist there, whether from a connected interface, a static route, or an IGP. A **null route advertisement** works for summary prefixes: a static route to a discard interface (null0) is configured for the aggregate, giving BGP a stable prefix to advertise even if the more-specific constituent routes flap underneath it. **Redistribution** pulls routes directly from another protocol — OSPF, EIGRP, or static — into BGP without requiring a match in the routing table. These three methods are not equivalent, and the method used matters for the next step in the algorithm, because it determines the origin code the route carries.

**AS_PATH length** is the first inter-AS attribute. BGP prefers the path with the **fewest** AS hops — the **shortest** AS_PATH. This is the most commonly understood BGP metric, but it is also easily manipulated. **AS_PATH prepending** is the technique of artificially inflating your own AS_PATH when advertising a prefix to a neighbor, making yourself appear farther away and less preferred. Prepending is how an organization influences **inbound** traffic — by making one entry point look shorter than another to external ASes.

**Origin code** is a tiebreaker that reflects how the route entered BGP. A route injected via the `network` command carries an origin of **IGP** (`i`), meaning it came from a connected or statically defined network. A route redistributed from another protocol carries an origin of **Incomplete** (`?`), meaning BGP cannot fully account for its origin. IGP is preferred over Incomplete. The third origin code, EGP, is historical and no longer used.

**MED** — the **Multi-Exit Discriminator** — is the mirror image of Local Preference. Where Local Preference controls which exit an AS uses for **outbound** traffic, MED is a suggestion to a neighboring AS about which entry point to use for **inbound** traffic. The **lowest** MED wins. MED is propagated to all routers within the neighboring AS, but it is not passed further — it only influences the immediate neighbor. Unlike Local Preference, MED crosses AS boundaries, but only one hop. If you have two links into your AS from the same upstream provider and you want traffic to enter through a specific one, you advertise a lower MED on that link.

After MED, the algorithm shifts from policy to protocol preference. BGP prefers **eBGP-learned paths over iBGP-learned paths** when both exist. Then it prefers the path with the **lowest** IGP metric to the BGP next hop — the path whose exit point is closest within the internal network, according to the IGP.

If paths are still equal after all of that, BGP reaches its final tiebreakers: the **oldest** active path (to prefer stability over change), then the path from the neighbor with the **lowest** Router ID, and finally the path from the neighbor with the **lowest** IP address. These last three steps are deterministic but arbitrary — they exist purely to ensure BGP always makes a consistent choice, not to encode any routing intent.

The algorithm matters because every BGP design decision maps to one of these steps. **Outbound** traffic shaping is Local Preference. **Inbound** traffic shaping is AS_PATH prepending or MED. Single-router overrides are Weight. Understanding which attribute to reach for — and which scope it affects — is what separates engineers who configure BGP from engineers who design with it.
