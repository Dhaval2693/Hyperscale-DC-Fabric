# BGP Message Types

Before understanding how BGP chooses paths, it helps to understand how BGP speaks. Unlike OSPF, BGP does not flood topology information or rely on multicast. It establishes a TCP session on port 179 first — and only then begins to exchange information. Everything BGP does across that session is built on exactly four message types.

The conversation begins with an **OPEN** message. When Router R1 connects to Router R2 over TCP, the first thing it sends is an OPEN. Inside that message, R1 declares its Autonomous System Number, the BGP version it supports (version 4 in modern networks), a proposed **Hold Time**, its **BGP Router ID**, and any optional capabilities such as multiprotocol support for IPv6 or VPNv4. It is an introduction and a negotiation in a single packet — R1 is saying what it is and asking whether R2 can work with those parameters. If R2 accepts, it replies with its own OPEN. If something does not match — wrong ASN, incompatible parameters, unsupported capability — the session does not proceed.

Once both routers have exchanged and accepted OPEN messages, **KEEPALIVE** messages take over. A KEEPALIVE carries no routing information at all. Its only purpose is to confirm that the neighbor is still reachable and the session should remain active. By default, a KEEPALIVE is sent every 60 seconds. The Hold Time negotiated during the OPEN exchange — typically 180 seconds — defines how long a router will wait before declaring the session dead. If that timer expires without receiving either a KEEPALIVE or an UPDATE, BGP tears down the session.

The actual routing information travels in **UPDATE** messages. An UPDATE can carry new route advertisements, withdrawals of previously advertised routes, or both at the same time. Alongside each advertised prefix, the UPDATE includes path attributes that describe the route — the **AS_PATH** showing which autonomous systems the route has traversed, the **NEXT_HOP** identifying where to forward traffic, the **LOCAL_PREF** used internally within an AS to prefer one path over another, and the **MED** used to signal a preferred entry point into an AS to external neighbors. BGP does not send the entire table in every UPDATE. It sends incremental changes — only what is new or what has been withdrawn since the last exchange.

When something goes wrong, BGP sends a **NOTIFICATION** message and immediately closes the session. There is no negotiation, no retry within the same session, no gradual degradation. A malformed message, a hold timer expiry, a capability mismatch — any of these triggers a NOTIFICATION, the session drops, and both routers must start the establishment process from the beginning.

These four messages — OPEN, KEEPALIVE, UPDATE, NOTIFICATION — are the complete vocabulary of BGP. Everything the protocol accomplishes, from forming a neighbor relationship to withdrawing a failed route, happens through one of them. Understanding the messages is the foundation.
