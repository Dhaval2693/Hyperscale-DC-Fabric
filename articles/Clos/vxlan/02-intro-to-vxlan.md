# VXLAN: Why the Data Center Needed a New Overlay

The problems that drove VxLAN's creation were not abstract. They were operational pain points that every large data center engineer was running into by the late 2000s, as virtualization changed the relationship between physical servers and the networks connecting them.

The first problem was **Spanning Tree Protocol**. Traditional data centers were built on Layer 2 networks — large flat broadcast domains that allowed virtual machines to live anywhere without IP changes. Spanning Tree was the protocol that kept those Layer 2 networks loop-free. But Spanning Tree does this by blocking redundant paths. In a data center with thousands of servers and hundreds of interconnected switches, that means a significant fraction of your physical links are sitting idle, blocked by Spanning Tree while you pay for them. Spine switches that could be providing forwarding bandwidth are instead waiting in standby. Convergence after a failure is measured in seconds, sometimes tens of seconds. And the fundamental scaling limit of Spanning Tree — that a single root bridge must have visibility into the entire domain — means it simply cannot manage a fabric of any real size.

The second problem was the **VLAN limit**. A VLAN identifier is a 12-bit field in the 802.1Q tag, which gives you 4,094 usable segments. For a small enterprise that is more than enough. For a cloud provider hosting thousands of customers, each needing their own isolated network, 4,094 is a ceiling you hit quickly. A single hyperscaler might need tens of thousands of isolated tenant segments. VLANs cannot provide that.

The third problem was **MAC address table scale**. Virtualization dramatically increased the number of endpoints that network switches needed to track. Where a physical server once had one MAC address, a virtualized server now had dozens — one per virtual machine. A top-of-rack switch serving 48 servers might need to track thousands of MAC addresses instead of hundreds. And as virtual machines migrated between physical servers for load balancing or maintenance, the MAC-to-port mappings changed constantly, forcing switches to relearn and invalidate cache entries. At scale, the MAC table churn became a significant problem.

**VXLAN** — Virtual Extensible LAN — was designed to address all three problems simultaneously. It was introduced in 2011 through IETF RFC 7348, developed by a consortium including VMware, Cisco, Arista, Broadcom, and others. The core insight was simple: instead of building large Layer 2 domains across the physical fabric, build a small, stable Layer 3 underlay and create the Layer 2 domains virtually on top.

VXLAN is a **MAC-in-UDP** encapsulation protocol. When a virtual machine sends an Ethernet frame, the VxLAN edge device wraps that entire frame — MAC header and all — inside a UDP packet and sends it across the IP underlay. The underlay sees ordinary IP traffic and routes it normally, using all available paths. The destination edge device receives the UDP packet, strips the VxLAN encapsulation, and delivers the original Ethernet frame to the target virtual machine. From the virtual machine's perspective, it is connected directly to a flat Layer 2 network. The physical fabric underneath has no idea.

The VLAN limit problem is solved by the **VNI** — the VXLAN Network Identifier. The VNI is a 24-bit field in the VXLAN header, which provides 16,777,216 unique segment identifiers. A cloud provider that was running out of VLAN headroom at 4,094 segments now has over 16 million to work with.

The Spanning Tree problem is solved by removing Spanning Tree from the picture entirely. The underlay is a routed IP network — it uses ECMP across all available links simultaneously, with no blocked ports and no root bridge. Convergence after a failure is driven by the routing protocol, not Spanning Tree, and happens in milliseconds.

The MAC table problem is mitigated by distributing the learning. Instead of every switch in the fabric needing to know every MAC address, only the edge devices — the VTEPs — need to track the MACs behind them. The fabric itself operates on IP addresses and routes to VTEP endpoints, not to individual virtual machines.

VXLAN uses **UDP port 4789** as the standard destination port. The source UDP port is set to a hash of the inner packet headers — source MAC, destination MAC, source IP, destination IP, and protocol — which distributes traffic across multiple ECMP paths in the underlay without requiring the underlay to inspect inside the encapsulation. The underlay treats each VTEP-to-VTEP flow as a distinct UDP flow and hashes it to a different physical path, preserving ECMP efficiency.

Understanding VxLAN requires understanding two fundamental components: the **VTEP** — the device that performs encapsulation and de-encapsulation — and the **VNI** — the segment identifier that determines which virtual network a frame belongs to.
