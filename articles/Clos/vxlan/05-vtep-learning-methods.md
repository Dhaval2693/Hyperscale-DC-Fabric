# How VTEPs Learn: From Flooding to MP-BGP EVPN

The previous article ended with a problem. The first time VM-A sends traffic to VM-B, Leaf 1's VTEP does not know which remote VTEP is hosting VM-B. It has to flood — send a copy of the encapsulated frame to every remote VTEP in the fabric. Only after that flood does it learn the answer, by observing the outer IP source address on the return traffic. In a small fabric with a handful of VTEPs, this is manageable. In a fabric with hundreds of leaf switches and thousands of VMs migrating between them, the flooding becomes a scalability problem.

This is the VxLAN control plane problem, and there are four main approaches to solving it. Each makes a different tradeoff between simplicity, scale, and dependency on other protocols.

**Static unicast tunnels** are the simplest approach and require no additional protocol. An engineer manually configures each VTEP with the IP addresses of all the other VTEPs it needs to communicate with. BUM traffic is replicated and sent as unicast to each configured peer — no multicast required, no control plane protocol required. This works reliably in small, stable environments where the set of VTEPs changes infrequently. The cost is operational: every time a new leaf switch is added to the fabric, every existing VTEP must be reconfigured. At a hundred leaf switches, that is one hundred manual configuration changes every time the fabric grows. Static tunnels do not scale.

**Multicast underlay** was the original approach described in the VxLAN RFC. Each VNI is mapped to a multicast group in the underlay. VTEPs that participate in a given VNI join the corresponding multicast group. When a VTEP needs to flood BUM traffic, it sends it to the multicast group — the underlay's multicast routing delivers it to all members efficiently, without the sending VTEP needing to know the IP addresses of its peers. Data-plane learning still happens: the sending VTEP's IP is visible in the outer header, so receiving VTEPs can add it to their mapping tables reactively. The multicast approach works well and eliminates the unicast replication overhead, but it requires PIM to be running in the underlay. Many modern data center fabrics are built as pure IP networks without multicast routing, making this approach impractical unless multicast is explicitly enabled — which adds complexity to the underlay that operators often prefer to avoid.

**MP-BGP EVPN** — Multiprotocol BGP with the Ethernet VPN address family — is the dominant approach in modern hyperscale data centers. It solves the flooding problem at the source by distributing MAC and IP reachability information through BGP before traffic flows, rather than learning it reactively from data-plane traffic.

The key insight of EVPN is that BGP is already running in the underlay to distribute VTEP loopback addresses. EVPN extends that same BGP session to also carry overlay information — specifically, which MAC addresses and IP addresses are behind which VTEP. When VM-A comes online behind Leaf 1, Leaf 1 advertises an EVPN route: "MAC AA:AA:AA:AA:AA:AA, IP 10.1.1.10, VNI 10001, reachable via VTEP 192.168.1.1." Every other VTEP in the fabric receives this advertisement and installs it in their mapping tables. When VM-B later tries to send traffic to VM-A, Leaf 2's VTEP already knows exactly where to send it — no flood required.

EVPN uses BGP route types to carry different kinds of information. **Type 2 routes** carry MAC and IP bindings — the core mapping between a virtual machine's identity and its VTEP. **Type 3 routes** carry inclusive multicast information — which VTEPs participate in a given VNI, used for BUM traffic handling. **Type 5 routes** carry IP prefix routes for inter-subnet routing, enabling the overlay to route between VNIs without hairpinning through a centralized gateway. This makes EVPN not just a MAC learning mechanism but a complete control plane for both Layer 2 and Layer 3 overlay services.

The integration with the underlay is elegant. The same BGP sessions used for underlay routing — spine and leaf peering over the Clos fabric — carry both the underlay VTEP reachability and the overlay MAC/IP bindings. There is no separate control plane protocol to deploy, no additional infrastructure to manage. BGP handles everything: routing the underlay, advertising VTEPs, and distributing tenant MAC and IP information across the fabric.

**VxLAN with LISP** takes a different approach, using the Locator/ID Separation Protocol as the control plane. LISP separates the concepts of location (where an endpoint is attached) from identity (who the endpoint is), which maps naturally onto the VxLAN VTEP-to-MAC problem. LISP is more common in campus networks than in data centers — its strength is handling endpoint mobility across large geographically distributed environments, where the mapping database is centralized rather than distributed. For data center fabrics, MP-BGP EVPN's distributed control plane and tight integration with the existing BGP underlay makes it the more natural choice.

The choice of control plane matters for more than just scalability. It determines how quickly new endpoints are discoverable, how gracefully the fabric handles VM migration, and how much BUM traffic the underlay carries under normal operation. A fabric using multicast or data-plane learning will flood unknown traffic across every VTEP. A fabric using MP-BGP EVPN learns endpoint locations before traffic flows and eliminates flooding entirely for known destinations. At the scale of a hyperscale data center — hundreds of leaf switches, hundreds of thousands of virtual machines — that difference in flooding behavior translates directly into usable fabric bandwidth.

MP-BGP EVPN is the control plane choice at META, Google, and other hyperscalers precisely because it does not require multicast in the underlay, scales linearly with the BGP control plane, supports both Layer 2 and Layer 3 overlay services in the same protocol, and integrates cleanly with the eBGP-based Clos underlay that these fabrics already run.
